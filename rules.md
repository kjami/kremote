# Project rules

Conventions and constraints for working in this repository. Aimed at humans and AI assistants both.

## Architecture

- **Single source of truth for device support**: `src/services/devices/IDeviceService.ts`. Every TV brand is a class that implements this interface ‚Äî no leaking brand-specific calls into screens or hooks.
- **`DeviceManager` is a singleton**. Don't instantiate it directly; use `DeviceManager.getInstance()`.
- **UI layer never imports from `services/adb` or `services/googletv` directly** ‚Äî only via `services/devices/*Service.ts` wrappers.
- **Hooks are the React boundary**. `useDevice` is the only hook that touches `DeviceManager`; screens consume the hook, never the manager.
- **Promise-coalescing for `connect()`**: services that can be called concurrently from rapid button presses (Sony, Google TV) must guard with an in-flight promise so we don't pile up parallel auth flows.

## TypeScript

- `strict: true`. No `any` unless interfacing with a typeless library; in that case isolate behind `as unknown as Foo`.
- Public service methods always return `Promise<T>` even when work is sync ‚Äî keeps the call sites uniform.
- Don't widen `RemoteKey` ad-hoc; add to the union type and update each device's command table.

## Storage

| Data | Where | Why |
|------|-------|-----|
| Device list (id, name, ip, type, port) | AsyncStorage | non-sensitive, fine in plaintext |
| `Device.authKey` (Sony PSK) | SecureStore (`device_authkey_<id>`) | shared secret |
| Sony auth cookie | SecureStore (`sony_auth_cookie_<host>`) | session credential |
| Polo client cert + private key | SecureStore (`gtv_client_cert_pem_v3`, `gtv_client_key_pem_v3`) | client identity |
| Polo paired flag | SecureStore (`gtv_paired_v3_<host>`) | trust state |
| Polo pinned server cert fingerprint | SecureStore (`gtv_pinned_cert_<host>`) | MITM check |
| ADB client RSA private key | SecureStore (`adb_private_key`, `adb_public_key`) | client identity |
| Per-device favorite app IDs | AsyncStorage (`fav_apps_<deviceId>`) | non-sensitive |
| Active device id | AsyncStorage (`active_device_id`) | non-sensitive |

**Rule**: anything that authenticates the user/device to the TV goes in SecureStore. When in doubt, default to SecureStore.

When you add a new SecureStore key, also add a deletion line to `DeviceManager.forgetDevice()` so the "Forget device" action keeps working.

## Networking

- **Only RFC1918 / link-local IPs**. The `isPrivateIp()` validator in `DeviceSelector.tsx` is the source of truth; reject anything else at form-submission time.
- **iOS ATS**: only `NSAllowsLocalNetworking: true`. **Never** add `NSAllowsArbitraryLoads`.
- **Android cleartext**: enabled globally via `expo-build-properties`, but the IP validator is the actual security boundary ‚Äî Android NSC can't express CIDR.
- **No third-party HTTP from the device**. We talk to the LAN only.
- **TLS server cert verification**: skipped (TVs are self-signed) ‚Äî but we **pin** the cert fingerprint after first pairing and refuse mismatches on reconnect.

## Crypto

- Use `node-forge` for RSA / SHA / cert generation. Don't add `react-native-rsa-native` back ‚Äî its Android `build.gradle` is broken with AGP 8.
- `react-native-get-random-values` is imported at the top of `App.tsx` to polyfill `crypto.getRandomValues`. Don't move that import.
- `Buffer` is polyfilled globally in `App.tsx`. Code below the entry point can use `Buffer.from(...)` freely.
- **1024-bit RSA is intentional** for the Polo client cert (Hermes is too slow at 2048). If you bump it back to 2048, also add a "generating key‚Ä¶" UI freeze indicator and warn that first pairing takes 20+ seconds.

## Logging

- **Never log**: PIN values, hex pairing codes after submit, raw cookie values, full RSA private keys.
- **OK to log**: status codes, response sizes, fingerprint prefixes (`fp.slice(0, 16) + '‚Ä¶'`).
- Use `console.log` for protocol traces, `console.warn` for recoverable errors, `console.error` for unrecoverable.
- All service-level logs use `[ServiceName] ...` prefixes (e.g. `[Sony]`, `[GoogleTV]`, `[ADB]`).

## Error handling

- **Surface errors to the user**. The `App.tsx` `handleKey`/`handleLaunch` wrappers catch and show errors as toasts ‚Äî keep that. Don't let `.catch(() => {})` swallow anything silently.
- All `expectXxx` / `connectXxx` flows should have a timeout (8‚Äì60s) so a hung TV doesn't hang the UI forever.
- **Pairing modals must be cancellable** ‚Äî `reject(new Error('‚Ä¶ cancelled'))` so the awaiting flow tears down cleanly.

## Commit conventions

- **One-line short summary** subject. No 70-character limit on rambling.
- **No `Co-Authored-By:` trailers.** Squash them out if a tool adds one.
- **Don't commit `node_modules/`, `ios/`, `android/`, `.expo/`, or `.claude/`** ‚Äî all are in `.gitignore`. The native dirs are regenerated by `expo prebuild` / EAS Build.

## Build & deploy

- **Local dev** (Mac with Android Studio / Xcode): `npx expo run:android` or `run:ios`.
- **Cable-free dev** (recommended): `eas build --profile development --platform android`, install the `.apk` from the EAS link, then run `npx expo start --dev-client` on the laptop.
- **Don't bump RN/Expo SDK casually** ‚Äî `react-native-tcp-socket` and the New Architecture compatibility need re-validation. New Architecture is currently disabled (`newArchEnabled: false`).
- After any change to `app.json` or native module set, **run `npx expo prebuild --clean`** locally before pushing to verify the manifest is valid.

## Adding a new device type

1. Add the type literal to `DeviceType` union in `src/types/index.ts`.
2. Create `src/services/devices/<Brand>Service.ts` implementing `IDeviceService`.
3. Register it in `DeviceManager.createService()`.
4. Add a button to the type picker in `DeviceSelector.tsx`.
5. If pairing is needed, register a new prompt callback (see how `setPinPrompt` / `setPairingPrompt` are wired).
6. Add a stub `listApps()` even if you can't fetch ‚Äî return `[]`.

## Adding a remote key

1. Add the key name to the `RemoteKey` union in `src/types/index.ts`.
2. Add an entry to **every** command table in `src/constants/commands.ts` (`SONY_IRCC`, `SAMSUNG_KEYS`, `FIRESTICK_KEYCODES`). TypeScript's `Record<RemoteKey, ‚Ä¶>` will fail compilation otherwise.
3. Wire a button in the relevant screen.

## Don't

- Don't introduce a global state library (Redux, MobX) ‚Äî `useDevice` is enough.
- Don't add a logger framework ‚Äî `console.*` is fine for an LAN remote app.
- Don't add Sentry / analytics without explicit user consent UI.
- Don't add a backend. The whole point is the app talks to TVs directly.
